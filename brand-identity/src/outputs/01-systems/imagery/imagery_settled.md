# Imagery — Settled

> [Produced via settle command — approved by human]

## Direction Selected

**A register system**, developed from `imagery_concepts_v1.md`. The three concepts do not represent competing alternatives from which one is selected — they identify three different operating conditions for imagery, all of which recur throughout the brand's life. The settled position governs when each applies.

A single-concept selection was evaluated and rejected: it would either over-restrict or under-govern. The register system is not a compromise. Each register is the correct answer under its own conditions and the wrong answer under the others. The discipline is in applying the governing conditions rigorously — not in selecting a preferred aesthetic.

The system is permanent, not transitional. All three conditions recur throughout the brand's life. Investor contexts recur. Non-specialist audiences recur. The condition "is the artifact mature enough to represent with honesty?" is a permanent governance question, not a stage-specific one.

**Register A — Specification Grade** (Concept 2 — The Mechanism): Primary operating register. Specification-grade technical diagrams where the artifact is mature and accurately representable, and the audience can read technical content.

**Register B — Structural Abstraction**: Secondary operating register. Geometric compositions derived from mechanism logic for contexts where artifact maturity or audience conditions do not permit Register A.

**Register C — Absent Image** (Concept 1 — The Absent Image): Tertiary operating register. Typographic system only, where it can carry the full communicative load without visual support.

**Concept 3 — The Authorization Moment**: Held. Not resolved. Not a fourth register — a subject question within the existing register system. Not pursued until the form question is answered.

## The Structural Problem This Module Solves

Krutho is simultaneously:

- A cryptographic identity primitive — technical and precise
- A standard-in-formation — not yet widely legible
- Pre-seed — without widespread artifact adoption
- Investor-facing first — before the mechanism is culturally readable

This creates a problem that cannot be resolved by choosing a single imagery direction: the system must communicate structural inevitability before the mechanism itself is fully legible, and it must do so honestly — without fiction, without overclaiming, and without borrowing the visual language of categories Krutho explicitly rejects.

The register system answers this. Three distinct modes of imagery operation, each governed by explicit conditions, each derivable from the same organising principle.

## Structural Properties Confirmed

### Register A — Specification Grade

**Governing condition:** The artifact is mature and accurately representable. The audience can read technical content. The context demands precision — showing the work is the communication goal.

**What it is:** Specification-grade technical diagrams. Certificate anatomy, chain-of-trust hierarchy, verification flow, data structure representations. The diagram shows how the standard works — not as explanation, but as evidence. Authority demonstrated through specificity.

**Formal grammar:**
- **Stroke:** Monolinear, consistent weight — derived from the wordmark's construction logic
- **Nodes:** Geometric, labeled in the system typeface (Spline Sans Mono for identifiers and data values, Spline Sans for structural labels), no icons
- **Composition:** Spatial logic follows mechanism structure — hierarchy expressed as hierarchy, chain expressed as directed sequence
- **Colour:** Achromatic by default. `color.action.default` (`#1D4A4B`) at structurally significant nodes only — root, issued state, verified terminus — following the UI colour rule. No colour-coding for differentiation.
- **Labels:** The label is the node's identity. "Human Principal" requires no person icon. Adding an icon is the primary drift signal.

**Cross-context behaviour:**
- Documentation and technical reference: primary context. The imagery system and the documentation system are the same system here.
- Developer-facing surfaces: diagrams appear where a claim needs substantiation — one diagram, precisely placed, as proof point. Not as hero.
- Social and distribution: a single precisely cropped diagram section — one node pair, one chain segment — reads as technical content rather than brand content.

**What breaks this:** Rounded nodes. Gradient fills. Icon-based systems. Colour-coded categories. Any diagram that could appear in a vendor pitch deck without modification. If the diagram looks like an enterprise security marketing explainer, it has drifted.

### Register B — Structural Abstraction

**Why this register exists:**

Register A requires two conditions to hold simultaneously: artifact maturity and audience technical literacy. Neither can be assumed at investor-facing, early-awareness, or non-specialist contexts — and both conditions recur throughout the brand's life regardless of stage.

Register B is not a workaround for immature artifacts. It is the correct visual register when the communication goal is legibility of system architecture and structural accountability, not technical depth. That goal recurs whenever the audience does not have — and cannot be assumed to have — familiarity with the specific standard.

The distinction between Register A and Register B is specificity, not quality. Register A shows the mechanism. Register B shows the logic of the mechanism, abstracted one level. Not metaphor — abstraction of structure, not meaning.

**Why not photography:**

Photography introduces human narrative, emotional tone, and cultural association. The hypothesis requires the brand to recede behind the specification. Photography foregrounds interpretation — it moves the identity toward story and brand world. This destabilises the structural thesis regardless of treatment.

**Why not diagrams alone:**

Diagrams are strong inside documentation. On surfaces where the communication goal is architectural legibility rather than technical instruction, diagrams read as explanatory — they imply teaching, they feel like vendor slides. Investors evaluating an infrastructure play are not reading the protocol. They are reading the architecture. Register B communicates architecture without teaching the mechanism.

**Why this is compatible with the hypothesis:**

The hypothesis requires authority demonstrated through precision and restraint. Register B is geometric, monolinear, non-decorative, non-metaphorical, and derived from system logic. It does not add warmth, brand personality, or narrative. It makes no claim the system cannot support. It is structurally honest about what it is: structural implication, not technical documentation.

**Governing condition:** The audience cannot be assumed to have technical familiarity with the specific standard, OR the artifact is not yet at a stage where specification-grade representation is accurate and honest. Applies at investor-facing contexts, non-specialist audiences, early-awareness moments, and any context where the communication goal is legibility of system architecture rather than technical depth.

**What it is:**

Compositions derived from mechanism logic, expressed without requiring technical decoding. The structural subject — accountability chains, layered architecture, verification flows, hierarchical relationships — is made spatially visible through geometric composition rather than diagrammatic representation. The image communicates system thinking; it does not document a specific mechanism.

The abstraction operates on relationships, not symbols:

| Mechanism Logic | Visual Abstraction |
|---|---|
| Root authority | Top-aligned primary block |
| Issued credential | Secondary connected module |
| Verification chain | Linear directional composition |
| Layer in stack | Stratified planes |
| Encoded fields | Modular grid blocks |

These are not decorative translations. Each must map to a real structural relationship in the mechanism. If a composition cannot answer "what mechanism relationship does this show?", it does not belong.

**Structural territories:**

- **Accountability chains:** Root → issued → verified. Hierarchy expressed as directed relationships. Authority originates at a specific node; the composition makes this spatially unambiguous. Not a network — a chain with direction and terminus.
- **Layered architecture:** Krutho occupies a layer, not the whole stack — a primitive within it. Vertical or depth-expressed compositions that make layer position legible without labeling it. Communicates: not a platform, not an app, a structural layer.
- **Modular field structure:** Structured fields, compositional logic, encoded relationships. Artifact-adjacent without literal representation. The visual logic of structured data, not the data itself.
- **Verification flows:** Deterministic path. Deliberate sequence. No branching, no mesh, no organic form. Straight geometry, defined start and terminus.

**Formal constraints:**
- **Geometry:** Monolinear strokes, hard edges, no rounded corners, no drop shadows — derived from wordmark construction logic
- **Colour:** Achromatic base throughout. `color.action.default` (`#1D4A4B`) at a single structurally significant point only — the same rule as Register A and the UI colour system. Never multiple accent colours. Never colour-coding for differentiation.
- **Density:** Sparse. Air between elements is structural. No decorative fill. No pattern. No texture. Each element earns its presence by belonging to the structural logic.
- **Placement:** Appear where structural emphasis is required. Do not fill backgrounds. Not used as texture or decoration. Every use requires a structural justification.
- **Scale:** Designed for the context. Not designed large and cropped.

**Drift signals:**
- Rounded or softened geometry — infographic / IAM vendor territory
- More than one accent colour — explainer diagram territory
- Organic or networked flows — AI visualisation territory
- Icon-based systems — vendor territory
- Gradient fills, glow effects — AI company aesthetic territory
- Pattern or texture use
- Any composition that could appear in a competitor pitch deck without modification
- Compositions that cannot be mapped to a real mechanism relationship

**Governance test:** Does this composition show structural logic derivable from the mechanism, or does it perform a visual aesthetic? If the latter, it is not Register B.

**The maturity arc:**

Register B is designed to recede as Register A becomes available. As the artifact matures and adoption grows, structured abstraction reduces and literal specification-grade representation increases. This trajectory reinforces the hypothesis — authority compounds with adoption, and the imagery system becomes more specific as the standard becomes more legible. Early stage: structural logic implied. Later stage: mechanism shown. Both are honest. Neither is a compromise of the other.

**Identity gate assessment:**

1. Does it read as infrastructure rather than a product? **Pass.** Layered stacks, hierarchical chains, and verification flows read as system architecture. They do not represent a company.
2. Does it signal technical credibility to a developer? **Pass.** Geometric, precise, non-decorative compositions are legible as engineered rather than designed.
3. Does it avoid implying Krutho must be present at verification? **Pass.** No imagery carries such an implication.
4. Would a technically skeptical developer trust it? **Conditional** — depends on execution precision. Correct formal discipline is required.
5. Could this be the identity of a cryptographic standard? **Conditional** — depends on whether formal constraints hold in execution. Drift toward infographic reads as vendor.
6. Does it demonstrate intellectual honesty? **Pass.** Register B makes no claim the system cannot support.
7. Does it exclude the wrong categories? **Pass**, provided formal constraints hold. Hard geometric forms, achromatic treatment, and monolinear construction exclude AI company and IAM vendor aesthetics when executed correctly.

### Register C — Absent Image

**Governing condition:** The typographic and spatial systems are strong enough to carry the full communicative load without visual support. The context is text-native by convention. No structural need for imagery can be identified.

**What it is:** No photographs. No diagrams. No compositions. The layout is designed with the assumption that no image zone will ever be filled.

Register C is not a fallback for contexts without available imagery. It is the correct answer when the typographic and spatial systems can hold without support — which in developer-native contexts (package documentation, technical reference, CLI output) is frequently the case.

**The marketing page test:**

Before deploying Register C at marketing scale, the typographic and spatial systems must demonstrate they can hold the above-the-fold treatment without visual support. This test must be conducted before Register C is adopted at any surface where Register B would otherwise apply. If the test fails, Register B applies.

If the test passes, Register C is the more structurally honest position — it makes no admission that imagery is necessary.

**What breaks this:** Adding any image "just for this context." The first exception ends the register. If an image is needed, the correct answer is Register B or Register A — not a compromise of Register C.

### Concept 3 — Held, Not Resolved

The Authorization Moment — imagery of the act by which a deliberate human decision becomes a credential — is structurally grounded. The brief's own structural observation establishes the human as the source of authority. Imagery that locates the moment that authority is exercised is evidence for a claim the mechanism makes.

The concept is held because the form of the subject is unresolved. The exploration brief in `imagery_concepts_v1.md` defines the question that must be answered: what does the authorization moment look like, and in what form — person, artifact, interaction trace — does it read as documentary evidence rather than brand communication?

If resolved, the Authorization Moment may belong to Register A (where the subject is a precise artifact) or to Register B (where the subject is a structurally derived composition of the moment). It does not constitute a fourth register — it is a subject question within the existing register system.

The exploration should be conducted if and when Concept 3 is needed. The form question must produce a definition specific enough to reject non-conforming imagery visibly before any production work proceeds.

### Register Coexistence and Governance

**How the registers coexist:**

All three registers are simultaneously valid and simultaneously present in the system. The question at any given context is which conditions hold — not which concept was selected.

- Documentation surface, mature artifact, developer audience: Register A.
- Legibility of system architecture to a non-specialist audience, or artifact not yet mature enough for honest specification-grade representation: Register B.
- Text-native context where the typographic and spatial systems can hold without support: Register C.

**Governing questions, in order:**

1. Is the artifact mature enough to represent with honesty at specification grade? If no — Register B (or Register C if text-native).
2. Can the audience be assumed to have the technical literacy to read specification-grade imagery? If no — Register B (or Register C if text-native).
3. Can the typographic and spatial systems carry the context without visual support? If yes — Register C. If no — Register B.
4. If conditions 1, 2, and 3 are all met — Register A.

**Drift signals:**

- **Register B appearing where Register A applies:** The artifact is mature, the audience can read it, but Register B is used instead — most commonly because Register A requires more production effort. This is drift by production convenience. The register must follow from conditions, not effort.
- **Register C deployed to avoid execution demands:** Register C at marketing scale before the marketing page test has been passed. Reads as incomplete, not controlled.
- **Register B drifting toward Register A:** Structural compositions become diagrammatic. This is not a failure — it is the maturity arc operating correctly. It should be welcomed, not resisted. The governance question is whether the diagram grammar applies.
- **Register A drifting toward Register B:** Diagrams simplified or rounded for accessibility. If an audience cannot read a specification-grade diagram, the correct answer is Register B, not a softened diagram. A diagram that violates the formal grammar is not a diagram — it is a failed attempt at both registers.

**What cannot happen:**

- Register A imagery appearing before the artifact is accurate and production-representative. Premature specificity reads as provisional and collapses credibility.
- Register B compositions appearing without a mappable mechanism relationship. If the composition cannot answer the governance test, it is decoration.
- Register C deployed at a context where the typographic and spatial systems have not been tested and confirmed to hold.
- Photography in any register, for any reason.

### Non-Negotiable Formal Properties

These apply across all three registers and are not subject to context-specific relaxation:

1. **No photography.** In any register, for any context, for any reason.
2. **No AI aesthetics.** Gradients, glows, generative visual language, neural network imagery — excluded across all registers.
3. **No IAM/enterprise security tropes.** Shields, locks, fortress imagery, dashboard design language — excluded across all registers.
4. **Achromatic foundation.** All imagery is achromatic by default. `color.action.default` (`#1D4A4B`) appears only at structurally significant single points, following the UI colour governance rule.
5. **Monolinear construction.** Consistent stroke weights. Derived from the wordmark's construction logic. No decorative weight variation.
6. **No rounded geometry.** Hard edges throughout. Rounded corners signal the wrong category.
7. **No icons.** Labels are the identity of nodes. Icons introduce symbolism; the system operates on structure.

**Single governance test:** Does this follow necessarily from the register conditions and the organising principle, or is it decoration? If the latter, it is absent.

## Constraints for Downstream Modules

**Layout**

The layout system must accommodate three operating conditions:

- Register C surfaces: designed with the assumption no image zone will ever be filled. No image zones reserved.
- Register B surfaces: named zones for structural compositions. Compositions do not fill backgrounds and are not used as texture — they occupy defined layout positions with structural justification.
- Register A surfaces: diagram zones within the spatial system. Diagrams respect the spatial system and do not float outside named zones.

The spatial system must be strong enough to hold above-the-fold marketing contexts without visual support before Register C can be deployed at that scale. The marketing page test is a layout system test as much as an imagery test.

**UI**

Register A is the native register for product surfaces — certificate display, credential structure, capability claims, API documentation, and onboarding. The diagram grammar applies to all UI-embedded technical diagrams without exception.

The colour governance rule applies inside all imagery: `color.action.default` is admitted only at structurally significant single points, following the same two-gate test as the UI colour system.

**Typography**

Spline Sans Mono deploys for identifiers, data values, and diagram node labels in Register A. Spline Sans for structural labels. The typeface system does not change at imagery boundaries — the diagram is an extension of the typography system, not a separate visual language.
